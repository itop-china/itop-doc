## 补丁

- Combodo 提供**补丁**包，只能使用此新功能安装。
- 补丁不包含完整的 iTop 包，而仅包含自上一个 iTop 版本（主要或次要）以来更改的文件。
- 补丁是一种配方，用于将一个 iTop 版本（源）修改为另一个 iTop 版本（微版本）。补丁的 iTop 源版本可以是主要或次要版本。该源版本是您的 iTop 必须运行的最低版本，才能安装该补丁。安装补丁时，如果补丁不适用于您的 iTop，则会显示错误消息。它不会破坏您的 iTop。
- 补丁旨在将您的 iTop 升级到特定的**微版本**。例如，假设 2.7.0-2 是一个微版本：如果您的 iTop 运行 2.7.0，则可以使用补丁包 `Patch-iTop-2.7.0-2.7.0-2.zip` 将其升级到 2.7.0-2。如果您的 iTop 已经运行 2.7.0-1，没问题，可以使用相同的补丁包升级到 2.7.0-2。

因为它从用户 PC 上传包到服务器，所以 Web 服务器必须允许上传此大小的文件。补丁包比完整包小得多：

- iTop 2.7.0 大约 20Mb
- iTop 2.7.0-1 的补丁大约 200Kb

### 限制

应用升级工具不会添加/删除模块/扩展，因此它**不能正确安装主要 iTop 升级**。如果误操作，只需运行 Setup 即可恢复情况。但通常，Web 服务器参数 `upload_max_filesize` 不够大，无法上传完整的 iTop 包。

## 使用

点击菜单**系统 / 应用升级**

页面将：

- 执行一些检查
- 允许您上传文件，其内容将被检查
- 允许您选择备份选项并执行升级

打开页面时的检查：

- Web 用户（您的 Web 服务器使用的用户）必须是所有者，或者必须对 iTop 根目录下的每个文件和目录（包括 iTop 根目录本身）具有**写入**访问权限。
- 它控制 `upload_max_filesize`（2 个变量）
- 它显示磁盘上的可用空间。

提供 zip 后的检查：

提供 zip 后，它会在临时目录中解压包并控制：

- 它是一个 zip
- zip 对应于已安装的产品（Community、Professional...）
- zip 符合清单，因此所有存在的文件都具有预期的大小和校验和，与离开 Combodo 工厂时一样
- zip 可以对应于补丁或完整包

然后：

- 将 iTop 置于维护模式（停止 cron 并防止用户修改数据）
- 执行所需的备份
- 从解压的包中复制 iTop 文件
- 编译 iTop（生成 env-prod 并更新数据库），与从 iTop Hub 安装或从 ITSM Designer 执行 Move to Production 的行为相同

如果这还不够，您将不得不手动执行安装过程，无论是新版本还是旧版本，如果您有无法修复的 XML 不兼容性。

## 常见问题

**问：我的 iTop 保持只读？**

**答：**如果升级过程在中间崩溃，可能需要手动删除 `<itop>/data/.readonly` 和 `<itop>/data/.maintenance` 文件。成功的 Setup 会在失败的应用升级后自动删除这些文件。

**问：我的升级失败但没有错误消息？**

**答：**检查 Web 服务器日志以了解原因。我们注意到，在执行备份时，当参数 `max_execution_time` 太低时，应用更新可能会失败。由于 iTop 被终止，无法显示任何错误消息。
